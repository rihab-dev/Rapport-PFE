\subsection{Description textuelle de cas d'utilisation « Demander accès via OAuth »}

\begin{table}[H]
\centering
\caption{Description textuelle « Demander accès via OAuth »}
\begin{tabular}{|p{4.5cm}|p{12cm}|}
\hline
\textbf{Nom du cas d'utilisation} & Demander accès via OAuth \\
\hline
\textbf{Acteurs} & Visiteur \\
\hline
\textbf{Préconditions} & Le visiteur n'a pas de compte dans le système\\
\hline
\textbf{Postconditions} & Une demande d'inscription est créée et en attente de validation par un administrateur\\
\hline
\textbf{Scénario principal} & 
\begin{minipage}[t]{\linewidth}
\begin{enumerate}[label=\arabic*., leftmargin=*]
\item Le visiteur accède à la page de connexion.
\item Le visiteur sélectionne le rôle souhaité (Client, Développeur, Chef de projet).
\item Le visiteur clique sur "Se connecter avec Google" ou "Se connecter avec GitHub".
\item Le système redirige le visiteur vers le fournisseur OAuth sélectionné.
\item Le visiteur s'authentifie et autorise l'application à accéder à ses informations.
\item Le fournisseur OAuth redirige le visiteur vers l'application avec un code d'autorisation.
\item Le système échange le code d'autorisation contre un access token.
\item Le système récupère les informations du profil utilisateur (email, prénom, nom, photo).
\item Le système vérifie si un compte existe déjà avec cet email.
\item Si aucun compte n'existe : le système crée une nouvelle demande d'inscription en attente de validation.
\item Le système envoie un e-mail de notification à tous les administrateurs actifs.
\item Le système envoie un e-mail de confirmation au visiteur.
\item Le système redirige le visiteur vers la page de connexion avec un message "Demande en attente de validation".
\end{enumerate}
\vspace{2pt}
\end{minipage} \\
\hline
\textbf{Scénarios alternatifs} & \begin{itemize}[nosep,leftmargin=0.8cm,topsep=0pt,itemsep=2pt, label=$\bullet$]
\item Compte existant actif : le système connecte directement l'utilisateur.
\item Demande déjà soumise : le système affiche "Votre demande est déjà en attente" sans créer de doublon.
\item Annulation OAuth : le visiteur annule l'autorisation, le système redirige vers la page de connexion.
\end{itemize} \\
\hline
\textbf{Scénarios d'exception} & \begin{itemize}[nosep,leftmargin=0.8cm,topsep=0pt,itemsep=2pt, label=$\bullet$] 
\item Email non disponible : le fournisseur OAuth ne fournit pas l'email, le système affiche un message d'erreur.
\item Fournisseur OAuth indisponible : le système affiche un message d'erreur technique.
\end{itemize} \\
\hline
\end{tabular}
\end{table}

\begin{enumerate}[label=\arabic*., leftmargin=*]
\item L'utilisateur accède au profil du client.
\item L'utilisateur consulte la section "Notes privées".
\item Pour ajouter une note : l'utilisateur clique sur "Ajouter", saisit le contenu et valide.
\item Pour supprimer une note : l'utilisateur clique sur l'icône de suppression et confirme.
\item Le système enregistre les modifications.
\item Le système affiche le message de succès.
\end{enumerate}
\vspace{2pt}
\end{minipage} \\

\textbf{Scénarios alternatifs} & Les membres de l'équipe (Développeur, Technicien) peuvent consulter (lecture seule) les notes privées du client dans la section détails du ticket pour avoir le contexte nécessaire au traitement. \\
\hline
\textbf{Scénarios d'exception} & L'utilisateur tente d'ajouter une note vide. \\
